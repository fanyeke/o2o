# 优惠券运营决策 Agent：方案草案 v0.1

## 1. 项目定位

本项目不把“天池 O2O 优惠券数据集”仅仅当作一个二分类建模题，而是把它改造成一个更贴近企业应用的系统：

> **基于优惠券历史行为数据，构建“经营监测 → AI 诊断 → 决策建议 → 人工审批 → 动作执行”的闭环系统。**

系统的核心不是单纯预测“券会不会被用”，而是：

* 哪些用户更值得被补贴；
* 哪类券效果差；
* 哪个商户的投券策略出现异常；
* 是否要缩小投放范围、调整券力度、暂停活动或发起复核；
* 如何将这些判断自动推送给运营人员，并允许其审批执行。

天池该赛题提供的是 **2016 年 1 月 1 日至 6 月 30 日的线上线下真实消费行为**，目标是预测 **2016 年 7 月领取优惠券后 15 天内是否使用**。官方评价使用的是按 `coupon_id` 分组后的平均 AUC。([天池][1])

# 2. 数据源与字段理解

## 2.1 数据集构成

可将数据理解为三类：

| 数据块    | 作用                |
| ------ | ----------------- |
| 线下训练数据 | 线下领券、无券消费、用券消费    |
| 线上训练数据 | 线上点击、购买、领券行为      |
| 线下预测样本 | 需要预测 15 天内核销概率的样本 |

官方预测输出的核心字段围绕：

* `user_id`
* `coupon_id`
* `date_received`
* `Probability`

其中 `Probability` 是参赛者需要给出的“15 天内用券概率”。这也说明官方 7 月样本本身是**无标签预测集**，不适合作为我们项目内部最终评测集。([天池][1])

---

## 2.2 线下核心字段

| 字段              | 含义            | 系统中的作用      |
| --------------- | ------------- | ----------- |
| `User_id`       | 用户 ID         | 用户实体        |
| `Merchant_id`   | 商户 ID         | 商户实体        |
| `Coupon_id`     | 优惠券 ID        | 优惠券实体       |
| `Discount_rate` | 折扣类型          | 券力度、补贴成本估算  |
| `Distance`      | 用户活动地点与最近门店距离 | 地理接近度特征     |
| `Date_received` | 领券日期          | 触发时间        |
| `Date`          | 消费日期          | 判断是否核销、核销时滞 |

`Discount_rate` 有两种形式：

* `x ∈ [0,1]`：直接折扣率；
* `x:y`：满 `x` 减 `y`。

`Distance` 表示用户常驻活动地点到最近门店的距离，单位是 `500 米` 的档位；`0` 表示小于 500 米，`10` 表示大于 5 公里。([天池][1])

官方对行为记录的解释是：

* `Date = null 且 Coupon_id != null`：领券但未使用；
* `Date != null 且 Coupon_id = null`：普通无券消费；
* `Date != null 且 Coupon_id != null`：优惠券消费。([天池][2])

---

## 2.3 线上核心字段

线上表比线下表多一个：

| 字段       | 含义                   |
| -------- | -------------------- |
| `Action` | `0=点击`，`1=购买`，`2=领券` |

线上表同样包含用户、商户、优惠券、折扣、领券日期、消费日期等字段，因此它可以作为用户线上活跃度、优惠券偏好、商户熟悉度等辅助特征来源。([天池][2])

# 3. 标签定义与项目中的训练集设计

## 3.1 监督学习标签

我们定义：

```text
label_redeem_15d = 1
当且仅当：
- 用户领取了优惠券；
- 后续发生了用券消费；
- 消费日期 - 领券日期 <= 15 天。
```

否则：

```text
label_redeem_15d = 0
```

这与官方任务“预测用户领取优惠券后 15 天内是否使用”一致。([天池][1])

---

## 3.2 推荐的时间切分方式

官方测试集本身没有标签，因此本项目内部评估建议使用历史数据重新切分：

| 数据段               | 用途          |
| ----------------- | ----------- |
| 2016-01 ~ 2016-04 | 训练集         |
| 2016-05           | 验证集         |
| 2016-06           | 测试集         |
| 官方 2016-07 样本     | 仅用于“上线模拟推理” |

这样做有两个好处：

1. 更贴近真实业务中的“用过去预测未来”；
2. 可以对模型、规则与 Agent 的决策进行离线评估。

# 4. 用 AIP 思路改造数据：从“表”变成“业务对象”

Palantir AIP 的核心不是单纯建表，而是将底层数据映射为：

* **Objects**：业务对象；
* **Properties**：对象属性；
* **Links**：对象关系；
* **Actions**：可执行动作；
* **Functions**：业务逻辑与计算函数。

Palantir 官方将 Ontology 定义为位于数据、虚拟表和模型之上的运营层；它通过对象、属性、链接表达业务语义，并通过 actions 和 functions 支持组织中的真实决策与执行。([Palantir][3])

我们不复制 Palantir 的私有实现，而是在 PostgreSQL / MySQL 中**复刻这种建模思想**。

# 5. 本项目中的轻量 Ontology 设计

## 5.1 原始层：Raw Layer

直接导入三张原始表：

```text
raw_offline_train
raw_online_train
raw_offline_test
```

这层只做：

* 原样保存；
* 字段类型修正；
* 缺失值规范化；
* 日期格式转换。

---

## 5.2 事件层：Event Layer

将原始数据统一转成业务事件。

| 事件表                       | 来源   |
| ------------------------- | ---- |
| `coupon_received_event`   | 领券   |
| `coupon_redeemed_event`   | 用券消费 |
| `ordinary_purchase_event` | 无券消费 |
| `online_click_event`      | 线上点击 |
| `online_purchase_event`   | 线上购买 |

这样做的好处是，后续 Agent 不需要理解原始表的混合语义，而是通过统一事件接口读取数据。

# 6. 业务对象之间的关系

| 关系                                                 | 说明        |
| -------------------------------------------------- | --------- |
| `User -> receives -> CouponReceipt`                | 用户领取优惠券   |
| `CouponReceipt -> belongs_to -> Coupon`            | 领取行为对应券   |
| `Coupon -> issued_by -> Merchant`                  | 券由商户发放    |
| `CouponReceipt -> may_generate -> DecisionCase`    | 某领券行为触发决策 |
| `DecisionCase -> has -> Recommendation`            | 决策案例产生建议  |
| `Recommendation -> approved_as -> ActionExecution` | 建议经审批后执行  |

这就是一个轻量版的业务语义网络。

# 7. 指标体系设计

这里要分三类：

1. 数据集可以直接计算的指标；
2. 需要模型估计的指标；
3. 企业里重要，但该数据集无法真实还原、只能模拟的指标。

---

## 7.1 可直接计算的指标

| 指标                            | 含义          | 用途        |
| ----------------------------- | ----------- | --------- |
| `coupon_receive_count`        | 领券次数        | 衡量曝光规模    |
| `coupon_redeem_count`         | 核销次数        | 衡量实际使用    |
| `coupon_redemption_rate`      | 核销数 / 领券数   | 衡量券投放质量   |
| `redeem_rate_15d`             | 15 天内核销率    | 与建模标签一致   |
| `avg_redeem_lag`              | 平均几天后核销     | 判断券刺激是否即时 |
| `merchant_coupon_redeem_rate` | 商户维度核销率     | 找异常商户     |
| `user_coupon_redeem_rate`     | 用户维度核销率     | 找高券敏感用户   |
| `distance_bucket_redeem_rate` | 不同距离档位核销率   | 判断地理触达效率  |
| `discount_type_redeem_rate`   | 不同券型核销率     | 比较折扣券与满减券 |
| `discount_depth_redeem_rate`  | 不同优惠力度下的核销率 | 分析让利是否有效  |

这些指标都能由原始字段和派生字段得到。([天池][1])

---

## 7.2 模型估计指标

| 指标                           | 含义        | 用途       |
| ---------------------------- | --------- | -------- |
| `pred_redeem_prob`           | 15 天内核销概率 | 模型核心输出   |
| `segment_expected_redeems`   | 人群预期核销量   | 用于活动规模评估 |
| `merchant_expected_redeems`  | 商户预计核销量   | 用于商户活动判断 |
| `coupon_expected_conversion` | 某券未来核销表现  | 用于券策略建议  |

---

## 7.3 需要模拟或外部补充的企业指标

### ROI

企业里常见定义可以写成：

```text
ROI = (增量收益 - 补贴成本) / 补贴成本
```

但本数据集没有直接提供：

* 订单金额；
* 毛利；
* 实际补贴成本；
* 未发券情况下的自然转化基线。

因此，**真实 ROI 无法由该公开数据集直接还原**。基于字段可知，数据中有折扣信息，但没有完整交易金额与利润信息。这里的 ROI 只能做成**可配置模拟指标**。([天池][1])

建议在系统里设计：

```text
simulated_gmv
simulated_margin_rate
estimated_subsidy_cost
estimated_incremental_profit
simulated_roi
```

---

### 推荐的模拟公式

```text
预计核销量 = Σ pred_redeem_prob

预计补贴成本 = Σ(pred_redeem_prob × 单券补贴估计)

预计增量收益 = Σ(pred_redeem_prob × 假定客单价 × 假定增量系数)

模拟 ROI = (预计增量收益 - 预计补贴成本) / 预计补贴成本
```

这里的：

* 假定客单价；
* 假定毛利率；
* 假定增量系数；

都做成后台配置项，而不是写死。

# 8. 决策逻辑：指标变化触发什么动作？

这部分是系统最“企业化”的地方。

## 8.1 决策类型一：商户券活动表现异常

| 条件                 | 含义 |
| ------------------ | -- |
| 近 7 日核销率显著低于近 30 日 |    |
| 发券量仍然很高            |    |
| 平均券力度不低            |    |

### 触发结论

> 可能出现“券发得多，但用户不买账”的低效活动。

### Agent 可给出的建议

* 缩小投放范围；
* 限定近距离用户；
* 降低低价值用户投放比例；
* 对该商户活动进入人工复核。

---

## 8.2 决策类型二：高折扣但低转化

| 条件               | 含义 |
| ---------------- | -- |
| 折扣力度高            |    |
| 核销率低             |    |
| 用户距离较远或历史对该商户低活跃 |    |

### 触发结论

> 补贴力度未转化为有效使用，活动可能存在浪费。

### Agent 建议

* 暂停该券型；
* 更换目标人群；
* 改成更低面额、更高匹配度的人群券；
* 推送运营审批。

---

## 8.3 决策类型三：高价值用户召回

| 条件              | 含义 |
| --------------- | -- |
| 用户历史核销率高        |    |
| 与商户距离近          |    |
| 最近一段时间领券但未消费    |    |
| 模型预测再次触达的核销概率较高 |    |

### 触发结论

> 用户可能处于“可挽回”状态。

### Agent 建议

* 给出召回券建议；
* 生成运营沟通策略；
* 进入审批后触发 Mock API 执行。

---

## 8.4 决策类型四：商户券型结构优化

| 条件            | 含义 |
| ------------- | -- |
| 满减券核销率持续低于折扣券 |    |
| 但补贴金额更高       |    |
| 同一商户中差异明显     |    |

### 触发结论

> 券型设计可能不合理。

### Agent 建议

* 优先改券型；
* 下调门槛；
* 或拆分为更轻量券策略。

# 9. 可配置规则方案

可以设计一份规则配置文件：

```yaml
decision_rules:
  merchant_campaign_underperform:
    enabled: true
    window_days: 7
    baseline_days: 30
    conditions:
      min_coupon_receives: 200
      redemption_rate_drop_ratio: 0.20
      avg_discount_depth_gt: 0.15
    action:
      create_case_type: merchant_campaign_alert
      call_agent: diagnose_campaign

  high_discount_low_redemption:
    enabled: true
    conditions:
      coupon_receive_count_gt: 100
      discount_depth_gt: 0.25
      redemption_rate_lt: 0.08
    action:
      create_case_type: coupon_strategy_review
      call_agent: propose_coupon_adjustment

  user_reactivation_candidate:
    enabled: true
    conditions:
      historical_redeem_rate_gt: 0.45
      pred_redeem_prob_gt: 0.70
      distance_bucket_lte: 2
    action:
      create_case_type: user_reactivation
      call_agent: recommend_followup
```

这样做的价值是：

* 指标阈值可调整；
* 规则可增删；
* Agent 不直接控制所有逻辑，而是在候选事件上做更灵活判断；
* 系统具备“状态机 + Agent”的混合形态。

# 10. Agent 在系统里到底负责什么？

Agent 不负责替代数据库，也不负责替代预测模型。

它主要负责：

## 10.1 证据整合

将以下信息汇总为“决策包”：

* 商户表现；
* 优惠券表现；
* 用户历史；
* 距离因素；
* 模型预测值；
* 规则触发原因。

---

## 10.2 方案比较

例如针对低效券活动，Agent 比较：

| 方案   | 可能效果   |
| ---- | ------ |
| 保持不变 | 继续低效发券 |
| 缩小人群 | 降成本    |
| 调整券型 | 提升转化   |
| 暂停活动 | 立即止损   |

---

## 10.3 输出审批建议

示例输出：

```json
{
  "decision": "recommend_adjust_campaign",
  "summary": "该商户近7日优惠券核销率较30日均值下降明显，且当前折扣力度偏高。",
  "evidence": [
    "近7日核销率显著走弱",
    "发券量仍处于高位",
    "高折扣券未带来同步转化提升"
  ],
  "recommended_action": [
    "暂停当前高折扣券",
    "优先对近距离高历史核销率用户投放",
    "一周后复评"
  ],
  "risk": [
    "缩减投放可能短期影响 GMV",
    "当前 ROI 为模拟估计，不代表真实财务结果"
  ],
  "need_human_approval": true
}
```

# 11. Agent 的工具接口设计

建议提供如下工具：

| 工具                                       | 功能             |
| ---------------------------------------- | -------------- |
| `get_user_profile(user_id)`              | 查询用户特征         |
| `get_merchant_profile(merchant_id)`      | 查询商户画像         |
| `get_coupon_profile(coupon_id)`          | 查询券画像          |

| `simulate_coupon_policy(params)`         | 模拟策略结果         |
| `create_decision_case(payload)`          | 创建决策案例         |
| `submit_action(action_payload)`          | 执行审批后的动作       |

这就形成了一个 Harness：

> **LLM + 工具 + 规则 + 数据对象 + 审批机制**

# 12. 仿照 AIP 的评估体系

Palantir 的 AIP Evals 支持：

* 创建测试案例；
* 定义评估函数；
* 比较不同 prompt 或模型版本；
* 做实验组合搜索；
* 查看失败案例；
* 因为 LLM 非确定性，官方还建议对 LLM 函数重复多次评估。([Palantir][4])

我们项目中可以做一个简化版：

## 12.1 决策评测集

每条样本包含：

* 输入案例；
* 预期动作；
* 不允许出现的错误；
* 是否必须升级人工审批。

---

## 12.2 评估指标

| 评估项                     | 说明           |
| ----------------------- | ------------ |
| `schema_validity`       | 输出 JSON 是否合规 |
| `evidence_coverage`     | 是否引用了关键证据    |
| `policy_compliance`     | 是否违反硬规则      |
| `action_reasonableness` | 建议是否合理       |
| `human_review_accuracy` | 是否在高风险时要求审批  |
| `consistency_score`     | 多次运行输出是否稳定   |

# 14. 最终完整流程

```text
1. 导入天池优惠券数据
2. 清洗并构建事件表
3. 生成 User / Merchant / Coupon / Receipt 业务对象

6. 指标异常触发 DecisionCase
7. Agent 拉取证据、比较方案、生成建议
8. 结果写入 PostgreSQL 和飞书多维表格
9. 飞书机器人推送审批卡片
10. 人工点击审批
11. 后端调用 Mock Action API 执行
12. 记录执行结果并进入评估
```
