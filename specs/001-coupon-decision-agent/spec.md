# Feature Specification: 优惠券运营决策 Agent 系统

**Feature Branch**: `001-coupon-decision-agent`  
**Created**: 2026-05-15  
**Status**: Draft  
**Input**: User description: "基于轻量业务 Ontology 与 Agent 的优惠券运营决策系统，实现从指标异常到审批执行的全闭环"

## User Scenarios & Testing

### User Story 1 - 运营人员接收异常预警并审批决策 (Priority: P1)

运营人员每日登录系统后，查看由系统自动触发的决策案例列表。每个案例展示商户/券/用户的异常指标（如核销率骤降、高折扣低转化等），并附有 Agent 自动生成的分析报告与建议方案。运营人员审阅证据后，可选择"批准"或"驳回"，系统据此执行或放弃对应动作。

**Why this priority**: 这是系统的核心价值闭环——从指标监测到人工决策再到执行，缺少此流程系统就是一个纯展示工具。

**Independent Test**: 可通过模拟一个商户核销率下降的决策案例，验证 Agent 生成建议、推送给运营、运营审批后 Mock Action 执行的完整链路。

**Acceptance Scenarios**:

1. **Given** 系统检测到某商户近7日核销率较30日基线下降超过20%，**When** 运营人员登录决策中心，**Then** 可以看到一条标记为"商户活动异常"的决策案例，包含触发指标数值、Agent 分析摘要和建议动作
2. **Given** 运营人员点开一条决策案例，**When** 查看 Agent 建议详情，**Then** 可以看到证据列表（至少3条）、风险提示、建议动作列表和置信度评分
3. **Given** 运营人员审批通过一条建议，**When** 点击"批准"按钮，**Then** 系统执行对应的 Mock Action，并记录执行结果
4. **Given** 运营人员驳回一条建议，**When** 点击"驳回"并填写理由，**Then** 案例状态更新为"已驳回"，不触发任何执行

---

### User Story 2 - 分析师查看商户/券/用户多维经营指标 (Priority: P1)

分析师通过系统 Dashboard 查看商户维度的关键经营指标：近7日核销率、30日基线、核销率变化幅度、平均折扣深度、活动健康分等。可以按商户、券型、时间段筛选，快速定位异常商户。

**Why this priority**: 指标计算是后续所有决策的基础，先让指标可见既可验证正确性，也方便人工判断。

**Independent Test**: 可通过导入测试数据后查看商户指标看板，验证每个商户的核销率、发券量等指标是否与手工计算一致。

**Acceptance Scenarios**:

1. **Given** 数据已导入系统，**When** 分析师查看商户指标看板，**Then** 每个商户显示近7日核销率、30日核销率、变化比例、发券量
2. **Given** 某商户近30日核销率计算完成，**When** 分析师查看该商户详情，**Then** 可以看到核销率趋势（按周或日）
3. **Given** 分析师需要对比不同券型，**When** 切换到券维度看板，**Then** 可以看到满减券 vs 折扣券的核销率对比

---

### User Story 3 - 系统自动触发规则扫描生成决策案例 (Priority: P2)

系统按日或按需运行规则引擎，扫描商户/券/用户指标。当指标满足预设阈值条件时（如核销率下降超过20%），自动创建 DecisionCase 并触发 Agent 进行诊断。

**Why this priority**: P2 是因为可以先手动触发案例验证 Agent 逻辑，规则自动化是价值增量而非必需品。

**Independent Test**: 可通过手动调用规则扫描 API 或命令，验证满足条件的商户是否被正确创建为决策案例。

**Acceptance Scenarios**:

1. **Given** 规则引擎配置了"商户核销率下降"规则（阈值：下降>20%，发券量>200），**When** 执行规则扫描，**Then** 符合条件的商户应生成一条 DecisionCase，状态为"待处理"
2. **Given** 某商户同时满足多条规则，**When** 执行规则扫描，**Then** 应为该商户生成多条独立的 DecisionCase
3. **Given** 规则扫描完成后，**When** 所有匹配的 DecisionCase 已创建，**Then** 系统自动触发 Agent 为每个案例生成建议

---

### User Story 4 - 历史决策案例检索与复盘 (Priority: P3)

运营人员可以按时间、商户、决策类型、状态等条件检索历史决策案例，查看每个案例的完整决策链路：触发指标→Agent 建议→审批记录→执行结果。

**Why this priority**: 复盘功能的价值依赖于前期积累的案例数量，前期案例少时意义不大。

**Independent Test**: 可通过创建若干测试案例并执行不同的审批操作，验证检索功能和详情展示是否正确。

**Acceptance Scenarios**:

1. **Given** 数据库中有多个历史决策案例，**When** 运营人员按商户 ID 搜索，**Then** 返回该商户相关的所有案例
2. **Given** 运营人员查看一个已完成的案例，**When** 展开详情，**Then** 可以看到触发时的指标快照、Agent 输出原文、审批人、审批意见和执行结果

---

### Edge Cases

- 当某商户完全没有优惠券发放记录时，指标计算如何处理（除零、空值）
- 当 Agent 调用 LLM 超时或返回格式错误时，系统如何降级处理（重试、使用默认建议、标记失败）
- 当审批回调因网络问题未到达时，案例状态如何保证最终一致性
- 当同一商户在同一天触发了多条规则，是否合并为一条决策案例还是各自独立
- 距离为 NULL 或距离为 0 的样本在计算距离相关指标时的处理差异

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 能导入天池 O2O 优惠券数据集（线下训练+测试），原始数据原样存储在 Raw 层
- **FR-002**: 系统 MUST 将原始数据清洗为统一事件表（领券事件、消费事件），存储在 Staging 层
- **FR-003**: 系统 MUST 每日自动计算用户、商户、优惠券维度的聚合指标（近7日/30日核销率、发券量、折扣深度等），存入 Feature 层
- **FR-004**: 系统 MUST 提供核销概率预测模型，基于历史数据预测领券后15天内核销的概率
- **FR-005**: 系统 MUST 支持可配置的决策规则（YAML 格式），定义指标触发条件和阈值
- **FR-006**: 系统 MUST 在规则触发时创建 DecisionCase 并触发 Agent 诊断流程
- **FR-007**: Agent MUST 至少调用2个数据工具查证后，再生成结构化建议输出（证据列表、建议动作、风险提示）
- **FR-008**: 系统 MUST 将 Agent 建议和工具调用轨迹持久化存储，用于审计和评估
- **FR-009**: 系统 MUST 提供 API 接口供飞书审批卡片回调，记录审批结果
- **FR-010**: 系统 MUST 在审批通过后执行对应的 Mock Action，记录执行结果
- **FR-011**: 系统 MUST 提供 REST API 用于查询决策案例、建议和指标数据
- **FR-012**: 系统 MUST 提供基于 Web 的 Dashboard，展示商户指标趋势和决策案例列表
- **FR-013**: 系统 MUST 支持按时间、商户、决策类型、状态检索历史决策案例
- **FR-014**: 指标计算与特征刷新 MUST 通过异步任务队列执行，不阻塞 API 请求

### Key Entities

- **User**: 优惠券的接收者，具有领券行为和消费行为。关键属性包括历史核销率、领券频次、距离倾向、线上活跃度
- **Merchant**: 发放优惠券的商户。关键属性包括核销率（7日/30日）、平均折扣深度、发券规模、活动健康分
- **Coupon**: 优惠券实体，具有类型（满减/折扣）、折扣深度、门槛金额、减免金额。与 Merchant 存在发放关系
- **CouponReceipt**: 领券事件，记录某用户在某天从某商户领取了某张券。是建模的原子粒度，包含距离、是否核销、预测概率
- **DecisionCase**: 决策案例，由规则触发，包含案例类型（商户异常/券策略复核/用户召回）、严重级别、触发指标快照、状态流转（待处理→已建议→已审批→已执行）
- **Recommendation**: Agent 输出的决策建议，包含决策摘要、证据列表、建议动作、风险提示、置信度评分、是否需要人工审批
- **ActionExecution**: 审批通过后执行的 Mock 动作记录，包含动作类型、请求参数、执行状态、执行时间

## Success Criteria

### Measurable Outcomes

- **SC-001**: 运营人员可在 3 次点击内完成从查看决策案例到审批执行的全流程操作
- **SC-002**: 核销概率预测模型 AUC 不低于 0.68（天池竞赛基线水平）
- **SC-003**: 从数据导入完成到商户指标看板可查看，时间不超过 30 分钟（含特征计算和指标聚合）
- **SC-004**: Agent 决策从触发到生成建议的时间不超过 30 秒（含工具调用和 LLM 推理）
- **SC-005**: 系统支持至少 3 种不同类型的决策规则并行运行（商户异常、高折扣低转化、用户召回）
- **SC-006**: Agent 生成的建议中，至少包含 3 条独立证据来支持其结论
- **SC-007**: 高风险建议（如暂停活动）必须标记为需要人工审批，低风险建议（如微调人群）可自动执行
- **SC-008**: 系统支持至少 5 个并发用户同时操作，响应时间在 2 秒以内

## Assumptions

- 数据集仅有 2016 年 1-6 月线下数据和 7 月预测样本，线上行为数据暂缺将影响特征丰富度
- 由于数据集没有真实交易金额和利润信息，ROI 只能作为可配置的模拟指标，不作为硬决策依据
- 系统 MVP 阶段使用 Mock Action 替代真实外部系统对接（如修改投放策略、发送优惠券等）
- 飞书审批卡片是运营交互的主入口，同时保留 Web Dashboard 作为辅助界面
- 系统用户（运营人员/分析师）具备基本的指标解读能力和决策判断能力
- 核销概率模型使用时间切分（1-4月训练、5月验证、6月测试）而非随机切分，以避免时间泄漏
- 每日指标刷新通过 Celery Beat 定时执行，可在配置中调整刷新频率
- LLM 调用失败时有重试和降级机制（最多重试3次，失败后标记案例为"诊断失败"）
